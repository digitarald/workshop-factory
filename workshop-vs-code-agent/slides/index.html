What Is Agent Mode and Why It Matters

Agent Mode is a built-in feature of GitHub Copilot in VS Code. To access it, open the Chat panel with ‚åÉ‚åòI (Mac) or Ctrl+Alt+I (Windows/Linux), then select Agent from the mode dropdown at the top of the panel.

You might already be familiar with Copilot Chat, which answers questions and suggests code snippets, or Inline Chat, which edits a single selection in your file. Agent Mode goes further ‚Äî it can autonomously create, edit, and delete multiple files to complete a task. Think of it as a pair programmer that doesn't just give advice but actually takes action: it reads your project, proposes a plan, writes code, and even runs terminal commands ‚Äî all from a single prompt.

What makes Agent Mode especially powerful is its iterative approach. After making changes, it checks for errors ‚Äî lint issues, compilation failures ‚Äî and self-corrects without you needing to re-prompt. For static web projects like the one you'll build today, this means the agent can scaffold an entire site structure in seconds.

You're always in control. Every change the agent proposes appears as a diff you can review before accepting. You can accept individual file changes, reject others, or undo everything.

To follow along today, make sure the mode picker at the top of the Chat panel shows Agent ‚Äî not Chat or Edit.

‚úì Check Your Understanding

How do you open Agent Mode in VS Code?

Reveal answer

Open the Chat panel with ‚åÉ‚åòI / Ctrl+Alt+I and select Agent from the mode dropdown at the top.

The Chat panel is your entry point for all Copilot interactions. The mode dropdown lets you switch between Chat, Edit, and Agent modes. Knowing where this control lives is essential for the rest of the workshop.

Name one key difference between Agent Mode and regular Copilot Chat.

Reveal answer

Agent Mode can autonomously create, edit, and delete multiple files, while regular Chat only provides text responses and code suggestions you must manually apply.

This is the core distinction: Chat talks, Agent acts. Agent Mode can read your workspace, create and modify files, and run terminal commands ‚Äî making it far more autonomous than a simple Q&A chat.

True or false: Agent Mode applies all changes to your files automatically without showing you what changed.

Reveal answer

False. Agent Mode shows proposed changes as diffs that you review and accept or reject before they are applied.

Staying in control is a key design principle. Every file creation, edit, or deletion appears as a diff you can inspect. You can accept individual changes, reject them, or undo everything. The agent proposes; you decide.

Anatomy of a Good Prompt

A well-crafted prompt is the difference between getting a useful first draft and spending rounds going back and forth. Structure your prompts around three key parts: what you want (the deliverable), how it should be built (constraints and style), and context about what already exists in your project.

Be explicit about file structure ‚Äî writing "Create index.html, styles.css, and script.js in the project root" produces far better results than "Make me a website." State your tech constraints up front: "Use only plain HTML, CSS, and vanilla JavaScript ‚Äî no frameworks, no CDN imports, no build tools." And describe the visual outcome you're after: "A single-page landing page with a dark hero section, a three-column features grid, testimonials, and a footer."

To ground the agent in your actual project rather than generic assumptions, use @workspace references. For example, writing @workspace/brief.md in your prompt tells the agent to read that file for project requirements. This is one of the most powerful features for getting tailored output.

A detailed, specific prompt may feel like more upfront effort, but it saves you multiple rounds of correction. And remember ‚Äî you can always iterate with follow-up prompts to refine the result.

‚ö° Prompt Agent Mode to Scaffold a Landing Page

Setup ~1 min

  1. Create a new empty folder called landing-page and open it in VS Code (File ‚Üí Open Folder).
  2. Create a file called brief.md and paste the starter content below into it. This gives the agent context about your project.

Write your prompt ~2 min

  1. Open the Chat panel (‚åÉ‚åòI / Ctrl+Alt+I) and select Agent from the mode dropdown.
  2. Type this prompt (feel free to customize the product name):
    Read @workspace/brief.md for the project brief. Create a single-page landing page with index.html, styles.css, and script.js. Include a sticky nav bar, a dark hero section with a CTA button, a three-column features grid, two testimonial quotes, and a footer with social links. Use only plain HTML, CSS, and vanilla JS ‚Äî no frameworks, no CDN links, no build tools. Make it mobile-responsive with a hamburger menu on small screens.
  3. Press Enter and watch the agent propose files.

Review the diffs ~4 min

  1. For each file the agent creates, review the diff:
    • Does index.html use semantic HTML5 elements?
    • Does styles.css avoid any @import of external stylesheets?
    • Does script.js use only vanilla JS (no require, no import)?
  2. Click Accept on files that look correct. Click Discard on any file that violates your constraints.
  3. If you discarded a file, send a follow-up prompt: "Regenerate styles.css without any external imports."

Test in the browser ~3 min

  1. Open index.html in your browser (right-click ‚Üí Open with Live Server, or double-click the file).
  2. Check: Does the page render? Do nav links smooth-scroll? Does the hamburger menu work at narrow widths?
  3. If something is broken, paste the browser console error into the Chat panel as a follow-up prompt.

Starter code ‚Äî brief.md

# Project Brief: Awesome App Landing Page

## Product
Awesome App ‚Äî a fictional productivity tool that helps you organize tasks.

## Sections needed
1. **Hero** ‚Äî large headline, subtitle, and a "Get Started" call-to-action button
2. **Features** ‚Äî three-column grid: "Fast", "Simple", "Secure" (use emoji for icons)
3. **Testimonials** ‚Äî two short customer quotes with names
4. **Footer** ‚Äî copyright line and placeholder social links

## Constraints
- Single page, no routing
- Plain HTML, CSS, vanilla JS only
- Mobile-responsive (hamburger nav on small screens)
- Dark hero section, light body
Hint 1

Make sure the mode dropdown at the top of the Chat panel says Agent ‚Äî not "Chat" or "Edit".

Hint 2

Include @workspace/brief.md in your prompt so the agent reads your project brief for context.

Hint 3

If the agent outputs code with a CDN link or framework import, discard that file and re-prompt with: "No external dependencies ‚Äî vanilla only."

Hint 4

You can accept or discard files individually ‚Äî you don't have to accept everything as a batch.

Hint 5

If the page looks broken after accepting, open browser DevTools (F12), copy the console error, and paste it as a follow-up prompt to the agent.

‚ö° Ask the Agent to Add a Feature to Your Page

Practice the iterative workflow by asking Agent Mode to add a new feature to the landing page you built in Module 2.

  1. Open the Chat panel (‚åÉ‚åòI / Ctrl+Alt+I) with Agent mode selected. Your previous conversation should still be visible ‚Äî the agent remembers context from earlier turns.
  2. Pick one of these follow-up prompts (or write your own):
    "Add a dark/light theme toggle button in the navbar. Store the user's preference in localStorage so it persists on reload. Use only CSS custom properties and vanilla JS."
    "Add a 'Back to Top' button that appears when the user scrolls past the hero section. Use vanilla JS and CSS only ‚Äî no libraries."
  3. Press Enter and let the agent propose changes.
  4. Review the diffs:
    • Does the agent modify existing files or create new ones?
    • Are the changes scoped to the feature you asked for, or did the agent touch unrelated code?
  5. Accept the changes, refresh your browser, and verify the feature works.
  6. If it doesn't work, describe the bug as a follow-up prompt ‚Äî e.g., "The toggle button doesn't switch colors. The CSS variables aren't being updated."

‚úì Key Takeaways

Name two things you should include in an Agent Mode prompt to get better results.

Reveal answer

Include specific constraints (e.g., "vanilla JS only, no frameworks") and a clear description of the desired output (e.g., file names, visual layout, required sections).

Agent Mode performs best when you remove ambiguity. Explicit constraints prevent the agent from making assumptions (like importing a CSS framework), and a clear deliverable description gives it a concrete target. Think of it like giving instructions to a new teammate ‚Äî the more context, the better the first draft.

After the agent proposes file changes, what should you do before accepting them?

Reveal answer

Review each diff to verify correctness: check that the code matches your constraints, uses no unwanted dependencies, and follows the structure you asked for.

Agent Mode is a productivity tool, not an autopilot. Every proposed change is a diff you should inspect ‚Äî just like reviewing a pull request. Check for unwanted dependencies, incorrect logic, and accessibility issues before clicking Accept.

How do you give the agent context about files that already exist in your project?

Reveal answer

Use @workspace references in your prompt ‚Äî e.g., @workspace/brief.md or @workspace/styles.css ‚Äî so the agent reads those files before generating code.

The @workspace reference is the key mechanism for grounding. Without it, the agent generates code based on generic patterns. With it, the agent reads your actual files and tailors its output ‚Äî matching your existing variable names, style conventions, and project structure.
